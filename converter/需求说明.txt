# 自动化规则转换与 Git 同步工具 (Rule-Converter) 需求文档

| 维度 | 信息 |
| --- | --- |
| **版本编号** | **V0.04 (Integrated Stable)** |
| **项目名称** | Rule-Converter |
| **主要目标** | 自动化规则抓取、格式转换、多路径分发及全量 Git 同步 |
| **运行环境** | Python 3.10+, Windows 10/11, Git for Windows |

---

## 1. 项目架构与路径规范

程序采用“根目录-项目-子目录”的三级架构，支持多任务路径解耦。

### 1.1 目录结构逻辑

* **Root Dir (根目录)**：所有 Git 仓库的物理存放路径（如 `D:\MyRules`）。
* **Project (项目名)**：对应 GitHub 仓库名，也是本地文件夹名（如 `CmccBannedList`）。
* **Sub-Dir (子路径)**：根据任务自定义，支持多级目录（如 `AI`）或根目录（`""`）。

---

## 2. 核心业务流程

### 2.1 步骤一：全量前置同步

在进行任何编辑或下载操作前，程序必须执行以下 Git 指令：

1. **`git fetch --all`**：获取远程所有分支的元数据更新。
2. **`git pull`**：将远程改动合并到本地，确保本地数据源（特别是本地仓库内的 `.list` 文件）是最新版本。

### 2.2 步骤二：增量转换逻辑

程序通过 `HEAD` 请求获取远程文件的 `ETag` 或 `Last-Modified` 标识符，并与本地 `config.json` 中的 `state` 对比。

* **若标识符一致且目标文件存在**：跳过该任务。
* **若标识符不一致**：执行下载与转换。

### 2.3 步骤三：转换格式规范 (LIST to CVC)

1. **首行标识**：文件起始行必须为 `payload:`。
2. **注释处理**：保留所有以 `#` 开头的行，并将其前置增加 **2 个空格** 缩进。
3. **规则过滤**：仅提取以 `DOMAIN,`、`DOMAIN-KEYWORD,`、`DOMAIN-SUFFIX,` 开头的行。
4. **去重逻辑**：对规则行进行**保序去重**，确保输出文件干净且无重复项。
5. **备份策略**：支持可选参数 `save_list`。若为 `True`，则在目标路径镜像保存原始 `.list` 文件。

### 2.4 步骤四：自动提交推送

若本次运行中有任何文件发生变动：

1. 更新 `config.json` 中的 `state`。
2. 执行 `git add .`。
3. 执行 `git commit -m "Auto-update rules @ [时间戳]"`。
4. 执行 `git push` 同步至远程仓库。

---

## 3. 配置文件规范 (`config.json`)

配置文件应包含两个核心模块：

### 3.1 Settings (静态配置)

* `root_dir`: 根路径。
* `project_name`: 仓库名。
* `git_integration`: 包含 `enabled`, `auto_pull`, `auto_push` 等开关。
* `user_agent`: 模拟浏览器标识，防止下载被拦截。

### 3.2 State (动态状态)

* 以任务名为 Key，存储对应的 `ETag` 或时间戳字符串。

---

## 4. 任务列表定义 (Tasks)

V0.04 预设了三个核心任务：

| 任务名称 | 来源 URL (Source) | 输出子路径 (Sub-Dir) | 备份 List? |
| --- | --- | --- | --- |
| **Gemini** | `.../list/Ai/Gemini.list` | `AI` | 是 |
| **OpenAI** | `.../list/Ai/OpenAI.list` | `AI` | 是 |
| **CmccBannedList** | `.../main/CmccBannedList.list` | `""` (根目录) | 否 |

---

## 5. 安全与认证要求

* **Git 认证**：必须在 Windows 凭据管理器中配置 GitHub 个人访问令牌 (PAT) 或配置 SSH Key。
* **路径安全**：程序内部路径处理需使用 `os.path.normpath` 和 `r"""` 原始字符串，避免 Windows 反斜杠转义错误。
